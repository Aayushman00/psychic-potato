# ADR-013: Keyboard Graph Abstraction for Movement Costs

**Status**: Accepted

**Context**
To prepare for a biomechanical movement model (Fitts’ law, joint‑level effort, etc.) we needed to decouple the *representation* of keyboard states from the *metric* used to compute movement costs. Previously the cost matrix `C` was built directly from Euclidean distances (or a simple placeholder). This made it hard to later replace the metric with a more sophisticated one that might involve intermediate “pass‑through” states or direction‑dependent costs.

**Decision**
Introduce a **KeyboardGraph** layer:
- Nodes = `KeyboardState` objects (physical key + modifier state).
- Edges = **primitive ergonomic actions**, defined by semantic changes in finger, hand, and modifier state (see `EdgeType`).
- Edge weights = placeholder numeric cost (to be supplied later by a biomechanical cost model).
- The movement cost matrix `C` is the **all‑pairs shortest‑path distance** on this graph.

This yields a clean separation:

```
KeyboardState (data) 
        → KeyboardGraph (topology + semantic edge types)
        → Edge‑weight assignment (biochemical model)
        → All‑Pairs Shortest Path → Movement Cost Matrix C
        → Objective (QAP) uses C
```

**Alternatives Considered**
1. **Directly compute C from a function of two states** – keeps the model simple but makes it impossible to express indirect paths (e.g., moving via a rest position) without expanding the function’s argument list.
2. **Pre‑compute a dense matrix and later edit entries** – would tightly couple the metric to the matrix representation, making later changes error‑prone.
3. **Embed the graph inside the Objective class** – would entangle the optimization algorithm with graph construction, reducing reusability.

We chose the explicit graph approach because it:
- Allows the graph topology to be edited independently of the cost function.
- Makes future work (e.g., adding directed edges, asymmetric costs, or waypoint nodes) localized to the graph module.
- Keeps the `Objective` class agnostic to how `C` is derived; it only consumes a numeric matrix.
- Enables sophisticated models like “cost to move from A to B via resting position” without altering the optimizer.

**Implementation Details**
- `KeyboardGraph` (in `src/core/keyboard_graph.py`) stores a list of `KeyboardState` nodes and an adjacency list where each edge carries an `EdgeType`.
- `build_semantic_keyboard_graph` creates a **complete** graph where each edge receives a semantic type based on the pair of states:
    - `MODIFIER_TOGGLE` if only the modifier flag differs (same finger & hand).
    - `HAND_SHIFT` if the hand differs.
    - `FINGER_CHANGE` if the hand is the same but the finger differs.
    - `SAME_FINGER` otherwise (same finger, hand, modifier).
- Edge weights are initially set to `1.0` (placeholder). The helper `set_edge_weights_by_type` lets a cost model map each `EdgeType` to a numeric cost.
- `KeyboardGraph.distance_matrix()` runs Floyd‑Warshall to produce all‑pairs shortest‑path distances.
- `kb_builder.build_cost_matrix` now delegates to `build_movement_cost_matrix`, which builds the semantic graph and returns its distance matrix.
- A optional visualizer `plot_keyboard_graph` (using networkx & matplotlib) draws nodes colored by finger and edges styled by their `EdgeType`.
