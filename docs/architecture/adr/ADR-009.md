# ADR-009: Neighbourhood operators for the Simulated Annealing optimiser

**Status**: Accepted

**Context**
The Simulated Annealing algorithm (ADR‑004) requires a set of moves that can traverse the space of injective mappings (assignments of piano notes to keyboard states) while allowing efficient computation of the change in objective (Δcost).  We designed three move types that are simple to implement, preserve feasibility, and give the solver both local and global exploration capabilities.

**Decision**
Define the following neighbourhood moves, each of which takes a current mapping `m: [0..N‑1] → [0..M‑1]` (list of length N) and returns a new mapping `m'`:

1. **Swap (two‑note exchange)**
   - Choose two distinct indices i, j uniformly at random.
   - Exchange their assigned keyboard states: m'[i] = m[j], m'[j] = m[i]; all other entries unchanged.
   - *Rationale*: Allows the exploration of arbitrary re‑assignments while affecting only two positions; Δcost can be computed by considering the four affected pairs (i,i), (j,j), (i,j), (j,i) and their interactions with all other indices.

2. **Block Shift (contiguous translation)**
   - Choose a start index s and a length L (2 ≤ L ≤ L_max, where L_max is a configurable fraction of N, default 0.2·N).
   - Choose a direction d ∈ {+1, –1} (forward or backward in pitch order).
   - For k = 0 … L‑1:
        - src = (s + k) mod N   (wrap‑around allowed to keep the operation cyclic)
        - dest = (s + k + d) mod N
        - Set m'[dest] = m[src]; all other positions remain as in m (effectively shifting the block by one keyboard index in the chosen direction).
   - *Rationale*: Captures the intuition that melodic fragments (scales, arpeggios) often translate together when transposed or when adjusting fingering; explores coordinated moves of multiple notes.

3. **Rotation (three‑note cycle)**
   - Choose three distinct indices i, j, k uniformly at random.
   - Perform a cyclic rotation: m'[i] = m[j]; m'[j] = m[k]; m'[k] = m[i]; all else unchanged.
   - *Rationale*: Enables the exchange of assignments among three notes without requiring a temporary empty slot; useful for escaping local minima where pairwise swaps are insufficient.

For each move type we compute the **Δcost** using the formula:

\[
\Delta = \sum_{u\in S}\sum_{v=0}^{N-1} W_{uv}\bigl[C_{m'(u),m(v)}-C_{m(u),m(v)}\bigr]
       +\sum_{u=0}^{N-1}\sum_{v\in S} W_{uv}\bigl[C_{m(u),m'(v)}-C_{m(u),m(v)}\bigr]
       -\sum_{u\in S}\sum_{v\in S} W_{uv}\bigl[C_{m'(u),m'(v)}-C_{m(u),m(v)}\bigr],
\]
where S is the set of indices altered by the move. This expression avoids double‑counting and runs in O(|S|·N) time; with |S| ≤ 3 for swap/rotation and ≤ L_max for block shift, the cost is effectively O(N) per move.

If the move violates the injectivity constraint (e.g., two notes would map to the same keyboard state), it is rejected immediately and a new candidate is drawn.

**Alternatives Considered**
1. **Single‑element random reassignment** – pick a note and assign it to a random free keyboard state. While simple, it often creates large disruptive changes leading to poor acceptance rates and slower mixing.
2. **Swap with a temporary “blank” slot** – requires maintaining an explicit unassigned state, complicating the representation.
3. **Arbitrary permutation of a random subset** – more expressive but significantly increases the cost of computing Δcost (O(|S|² · N)) and complicates detailed balance proofs.
4. **Adjacent‑only swaps (only neighbours in pitch order)** – too restrictive; prevented the explorer from reaching distant, potentially better configurations in early tests.

**Consequences**
- **Positive**: 
    - The move set is ergodic (any permutation can be reached by a sequence of swaps).
    - Δcost computations are cheap, enabling tens of millions of Monte Carlo steps per second on a modern CPU.
    - The three move types together give a good balance of local refinement (swap/rotation) and broader exploration (block shift).
- **Negative**: 
    - The block‑shift move assumes a linear ordering of notes (pitch); for non‑octave‑preserving transformations this may be less effective, but it remains useful as a heuristic.
    - No move explicitly models finger‑specific constraints (e.g., avoiding thumb‑under on black keys); however, such constraints are implicitly encoded in the C matrix (finger‑change and posture terms) and will be respected by the cost‑based acceptance criterion.
- **Implementation Impact**: 
    - Created `src/core/optimiser/neighbourhood.py` exposing functions: `random_swap(m)`, `random_block_shift(m, L_max)`, `random_rotation(m)`, and corresponding `delta_*` functions that accept the objective instance and return Δcost. 
    - The main simulated annealing loop (in `simulated_annealing.py`) selects a move type with configurable probabilities (default: swap 0.5, block‑shift 0.3, rotation 0.2) and applies the chosen operator.
