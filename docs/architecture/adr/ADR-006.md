# ADR-006: Repository and directory structure

**Status**: Accepted

**Context**
As the project evolved from exploratory scripts to a reusable codebase with accompanying documentation, a clear layout was needed to separate source code, tests, experiments, and the extensive research/architecture documentation required by the process.

**Decision**
Adopt the following structure:

```
project/
├── docs/                 # All documentation (research, architecture, README)
│   ├── research/         # Literature review, methods, formulation, results, etc.
│   └── architecture/    # High‑level design, ADRs, data flow, etc.
├── src/                  # Source code (Python packages)
│   ├── core/             # Fundamental building blocks (weight & cost matrices, objective, optimiser)
│   ├── cli/              # Command‑line interface entry point
│   ├── evaluation/       # Metrics, plotting, user‑study helpers
│   ├── calibration/      # Scripts to run the Fitts‑based calibration
│   └── config/           # Default configuration files (JSON)
├── tests/                # Unit and integration tests (pytest)
├── experiments/          # Output from runs (not committed)
├── requirements.txt
├── setup.py
└── README.md             # Brief project summary (mirrors docs/README.md)
```

**Alternatives Considered**
1. **Flat layout** – all .py files in the project root. Quick for early prototyping but becomes unmanageable as the number of modules grows.
2. **src/lib/ and src/apps/** separation – introduced unnecessary indirection for a primarily library‑style project with a single CLI.
3. **Placing documentation inside the repo root (e.g., README, CONTRIBUTING)** – would mix user‑facing docs with in‑depth research notes, making it harder for newcomers to locate the appropriate material.
4. **Using a separate wiki or external site for documentation** – defeats the requirement that all research decisions be persisted within the repository for reproducibility.

**Consequences**
- **Positive**:
    - Clear separation of concerns makes navigation straightforward for developers and reviewers.
    - Researchers can locate the exact document that captures a decision (e.g., `docs/research/05-design-evolution.md`).
    - Build tools (pytest, sphinx, etc.) can be scoped to the appropriate directories.
    - The explicit `experiments/` folder keeps ephemeral runtime outputs out of version control while still being accessible for analysis.
- **Negative**: Slightly longer relative paths when importing modules (e.g., `from src.core.kb_builder import KeyboardBuilder`), but this is mitigated by configuring the Python path via `setup.py` or using an editable install (`pip install -e .`).
- **Implementation Impact**: No extra code changes required; essentially a matter of moving existing files into the prescribed locations and updating import statements accordingly. The structure has been in place since the early stages of the implementation and has not impeded development.
