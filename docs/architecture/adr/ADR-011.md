# ADR-011: Separation of Concerns between Objective and Mapping

**Status**: Accepted

**Context**
During the refactor of the optimization engine, we recognized that the original design mixed evaluation and mutation responsibilities within the `Objective` class. This tight coupling made the evaluator stateful, hindered reuse in alternative optimization strategies, and complicated verification of delta correctness. To improve modularity, testability, and flexibility, we decided to separate concerns: the `Objective` class becomes a pure evaluator (stateless apart from cached score), while all mutation logic resides in the `Mapping` class.

**Decision**
- `Objective` (in `src/core/objective.py`) is responsible only for:
  - Storing references to the weight matrix **W** and cost matrix **C**.
  - Providing a `score` property that computes the Quadratic Assignment Problem objective \(\sum_{i,j} W_{ij} C_{f(i),f(j)}\) on demand.
  - Providing a `delta(move_type, *args)` method that returns the exact change in objective for a given neighbourhood move (swap, block shift, rotation) using the static helper `_delta_from_arrays`.
  - Providing an `apply_move(move_type, *args)` method that delegates mutation to the appropriate `Mapping.apply_*` method and updates the cached score using the corresponding delta.
- `Mapping` (in `src/core/mapping.py`) is responsible for:
  - Holding the current assignment list `assignment` and the number of keyboard states `M`.
  - Implementing immutable mapping operations (`copy`, `to_list`, etc.).
  - Providing mutator methods:
    - `apply_swap(i, j)`
    - `apply_block_shift(start, length, direction)`
    - `apply_rotation(i, j, k)`
  - Ensuring that `M` is set before any mutator is called.
- The optimizer (e.g., Simulated Annealing) interacts with both abstractions:
  - It calls `objective.delta(move_type, *args)` to evaluate a candidate move.
  - If the move is accepted, it calls `objective.apply_move(move_type, *args)` to mutate the state and update the score.
  - It never directly manipulates `mapping.assignment` or calls `Mapping` mutators except through `objective.apply_move`.

**Consequences**
- **Positive**:
  - `Objective` is now pure and easily reusable in other optimization contexts (e.g., greedy, tabu search, parallel tempering) without modification.
  - Mutation logic is centralized in `Mapping`, making it easier to verify correctness (e.g., via property‑based tests that compare `delta` against full recomputation).
  - The clear separation simplifies reasoning about thread‑safety: multiple optimizers can share the same `Objective` instance (read‑only) as long as they operate on distinct `Mapping` instances or synchronize access.
  - Delta correctness can be verified independently of mutation implementation.
- **Negative**:
  - Slight increase in delegation overhead (one extra method call per move), negligible compared to the objective computation.
  - Requires disciplined use of the wrapper `apply_move` to keep the cached score consistent; direct use of `Mapping` mutators bypasses the score update.
- **Implementation Impact**:
  - Modified `src/core/objective.py` to remove internal mutable mapping and score fields, introduce `_delta_from_arrays` static method, and delegate mutation.
  - Updated `src/core/mapping.py` to ensure `M` is properly initialized and to keep mutator methods unchanged (they already existed).
  - Updated ADR‑004 and ADR‑007 to reflect the new responsibilities.
  - Added property‑based tests in `tests/test_objective_delta.py` to validate delta correctness.
