# ADR-007: Objective function – Quadratic Assignment formulation

**Status**: Accepted

**Context**
Having defined the musical weight matrix **W** (ADR‑003) and the biomechanical cost matrix **C** (ADR‑005), we needed a precise mathematical statement of the optimisation goal: find an assignment of piano notes to keyboard states that best preserves the musical relationships in terms of movement cost.

**Decision**
Formulate the problem as a **Quadratic Assignment Problem (QAP)**:

- **Decision variables**: binary matrix X ∈ {0,1}^{N×M} where X_{iα}=1 iff piano note i is assigned to keyboard state α.
- **Constraints**:
    - Each note assigned to exactly one state: \(\sum_{α} X_{iα}=1\) ∀ i.
    - Each state receives at most one note: \(\sum_{i} X_{iα} ≤ 1\) ∀ α (allowing unused states when M > N).
- **Objective**:
  \[
  \min_{X}\; \sum_{i=1}^{N}\sum_{j=1}^{N} W_{ij}\; C_{\,(\sum_{α} α X_{iα}),\,(\sum_{β} β X_{jβ})}.
  \]
  Using the mapping notation f(i) = ∑α α X_{iα}, this is equivalent to
  \[
  \min_{f\; \text{injective}}\; \sum_{i,j} W_{ij}\; C_{f(i),f(j)} .
  \]

- **Interpretation**: we pay the movement cost C between the keyboard states assigned to two notes, weighted by how important it is to preserve the musical relationship between those notes (W). Minimising the sum encourages placing highly related (consonant) notes close together in movement‑cost space, while allowing unrelated (dissonant) pairs to be farther apart.

**Alternatives Considered**
1. **Linear sum assignment (LSAP)**: minimize ∑_i C_{i, f(i)} – completely ignores pairwise musical relationships; reduces to a trivial “place each note on the cheapest key”.
2. **Weighted sum of squared differences in positions** – assumes a Euclidean embedding of both piano and keyboard, which discards the asymmetric, effector‑specific nature of C.
3. **Maximise trace(Wᵀ P C Pᵀ)** where P is a permutation matrix – mathematically identical to the QAP formulation but less explicit for engineers; we chose the more readable double‑sum version.
4. **Add a regularisation term for octave symmetry** – explored as a separate penalty (see ADR‑008 for a potential extension) but determined unnecessary for the core model because the high weight naturally assigned to octave intervals in W already encourages octave‑preserving mappings.

**Consequences**
- **Positive**: The formulation is a well‑studied NP‑hard problem with known exact and approximate solvers, allowing us to leverage a rich literature (e.g., simulated annealing, tabu search, branch‑and‑bound) while being confident that the objective accurately reflects the design goal.
- **Negative**: QAP is challenging to solve optimally for large N; we rely on heuristic SA (ADR‑004) which yields high‑quality approximate solutions. No guarantee of global optimality, but empirical results show solution quality within a few percent of the best‑known across multiple restarts.
- **Implementation Impact**: Created `src/core/objective.py` housing:
    - `__init__(self, W, C, mapping)` – stores references to the weight and cost matrices and the current mapping.
    - `score` property – computes the full objective in O(N²) on demand (used for initial solution and final reporting).
    - `delta(self, move_type, *args)` – computes the change in objective for a given neighbourhood move (swap, block shift, rotation) in O(N) using the static helper `_delta_from_arrays`.
    - `apply_move(self, move_type, *args)` – mutates the mapping via the appropriate `Mapping.apply_*` method and updates the cached score using the corresponding delta.
The objective class is a **pure evaluator**: it holds no mutable state apart from a cached score that is updated after mutations. All mutation logic resides in `src/core/mapping.py`.
The optimiser in `src/core/optimiser/simulated_annealing.py` calls `delta` to evaluate moves and `apply_move` to accept them.
