# ADR-004: Use Simulated Annealing as primary optimiser

**Status**: Accepted

**Context**
The problem to solve is a Quadratic Assignment Problem (QAP) with N≈61 (MVP) or N≈88 (full keyboard) facilities and M keyboard locations (M≥N). QAP is NP‑hard; exhaustive search is infeasible. We required a heuristic that:
- Works with a discrete permutation/injective mapping representation.
- Allows easy integration of problem‑specific neighbourhood moves (swap, block‑shift, rotation).
- Needs only the ability to compute the objective function (or its Δ‑cost) – no gradients.
- Provides a mechanism to escape local minima while being simple to implement and tune.

**Decision**
Adopt **Simulated Annealing (SA)** as the core optimisation algorithm.
- State representation: a length‑N list `mapping[i]` giving the index of the keyboard state assigned to piano note i.
- Neighbourhood moves (see ADR‑009):
    1. **Swap** two randomly selected notes.
    2. **Block shift** a contiguous segment of notes in pitch order by one keyboard position forward or backward.
    3. **Rotation** of three distinct notes’ assignments.
- Acceptance follows the Metropolis criterion: `P = exp(-Δcost / T)` if Δcost > 0, else 1.
- Temperature schedule: geometric cooling `T_{k+1} = α * T_k` with α = 0.95, starting temperature T₀ set so that a worsening move of magnitude equal to the 95th percentile of observed |Δcost| is accepted with probability ≈ 0.8.
- Terminate when temperature drops below `T_min = 1e-4` or after a fixed number of temperature levels without improvement.
- Perform multiple independent runs (R = 30) with different random seeds and retain the best solution found.

**Alternatives Considered**
1. **Tabu Search** – strong local search capabilities but requires careful tabu‑list management and aspiration criteria; more parameters to tune.
2. **Genetic Algorithm** – needs encoding of permutations and crossover operators that often produce infeasible offspring (duplicate or missing keys); repair operators add complexity.
3. **Greedy / Iterative Improvement** – tends to get trapped in poor local minima given the rugged QAP landscape; solution quality substantially worse than SA in preliminary tests.
4. **Exact branch‑and‑bound** – guarantees optimality but exponential time and memory; not viable beyond N≈20.
5. **Spectral relaxation / SDP bounds** – provide lower bounds but delivering a feasible integer solution requires costly rounding and post‑processing; not as straightforward for our asymmetric C.

**Consequences**
- **Positive**: SA is straightforward to implement, needs only the objective and Δ‑cost functions (already available from the QAP formulation). It yields solution quality within a few percent of the best‑known solutions after modest computation (seconds for MVP, tens of seconds for full keyboard on a standard laptop). Multiple restarts reduce variability.
- **Negative**: Being stochastic, SA does not guarantee optimality; however for engineering purposes a near‑optimal solution that meets ergonomic and musical criteria is sufficient.
- **Implementation Impact**: Added module `src/core/optimiser/simulated_annealing.py` and neighbourhood module `src/core/optimiser/neighbourhood.py`. The optimiser is decoupled from move generation: it only requires an `Objective` instance capable of computing `delta` for a given move and a `Mapping` instance that provides `apply_*` methods to mutate the state. Configuration of SA hyper‑parameters is stored in `src/config/sa_params.json`.
