# ADR-002: Represent keyboard as weighted directed graph

**Status**: Accepted

**Context**
To model the biomechanical cost of moving between keys we need a metric that captures direction‑dependent factors (e.g., finger change, hand change, modifier activation, posture). An undirected weighted graph would assume symmetric cost, which is inconsistent with measured asymmetries such as index→middle versus middle→index finger transitions.

**Decision**
Define the keyboard graph **Gₖ = (Vₖ, Eₖ, C)** where:
- Each element of Vₖ corresponds to a distinct **keyboard state**: a physical key label together with the hand, finger, and modifier layer used to depress it.
- For every ordered pair (a,b) ∈ Vₖ × Vₖ we place a directed edge with weight C_ab = predicted movement time (ms or s) from state a to state b, computed via the Fitts‑type + finger/hand/modifier/postural model described in ADR‑005.
- Self‑loops (a→a) have weight equal to the basic actuation cost c_0 (key down/up without lateral movement).
The graph is thus **complete** (or densely connected within a practical neighbourhood radius) and **asymmetric**.

**Alternatives Considered**
1. **Undirected weighted graph** – would force C_ab = C_ba, losing empirically observed asymmetries; preliminary tests showed resulting layouts placed notes in ways that felt awkward when actually performed (e.g., favoring moves that are cheap in one direction but expensive in the reverse).
2. **State‑augmented graph with separate layers for each finger/hand** – increases node count significantly without adding expressive power beyond what the (hand,finger,modifier) tuple already captures.
3. **Cost as a function on the fly** – would increase per‑move evaluation complexity; pre‑computing the matrix yields O(1) look‑ups and enables fast Δ‑cost calculations.

**Consequences**
- **Positive**: Captures direction‑specific biomechanical realities; enables the optimiser to distinguish, for example, using the index finger to move to a neighboring key versus using the pinky. Asymmetry also allows the model to penalize awkward hand‑crossing motions appropriately.
- **Negative**: Increases storage from O(M) to O(M²); however M (number of keyboard states) remains modest (~200‑400 after pruning), making a dense matrix feasible (< 2 MB). If scaling to very large custom keyboards becomes necessary, a sparse representation can be adopted without affecting the optimisation logic.
- **Implementation Impact**: The keyboard builder (see `src/core/kb_builder.py`) now emits a matrix `C`. The QAP objective uses `C[f(i), f(j)]` directly; no symmetrisation is performed.
