# ADR-012: Keyboard States instead of Physical Keys

**Status**: Accepted

**Context**
The original design treated each physical keyboard key as a discrete assignment target for piano notes. This approach failed to capture the expressive potential of modifier keys (Shift, Alt, etc.) that musicians naturally use to expand the effective key‑set without increasing hardware complexity. Moreover, the optimizer was implicitly minimizing the number of distinct physical keys used, which is not the goal—ergonomic playability is.

**Decision**
Model the optimization domain as a set of **keyboard states**, where a state is a tuple *(physical key, modifier state)*. The optimizer assigns each piano note to a distinct keyboard state; multiple states may share the same underlying the same physical key (e.g., “Q” and “Shift+Q”). This decouples the notion of a physical key from the combinatorial choice available to the pianist.

**Alternatives Considered**
1. **One‑hot modifier flags per key** – keep the physical‑key list and add Boolean modifiers as separate dimensions in the cost matrix. This would keep the assignment matrix size unchanged but require a more complex cost function that depends on modifier combinations; it also obscures the discrete nature of the assignment.
2. **Separate layers as independent keyboards** – create multiple keyboard copies (one per modifier) and run independent optimizations, then merge. This would lose cross‑modifier coupling (e.g., avoiding placing two frequently simultaneous notes on the same shift‑state key) and increase implementation complexity.
3. **Explicit choke‑set constraints** – forbid assigning two notes to states that require the same physical key with incompatible modifiers (e.g., both Shift and AltGr). This adds constraints but does not enrich the search space; the optimizer would still be limited to the raw key count.

We chose the explicit state‑enumeration approach because it:
- Is simple to implement: the keyboard becomes a list of State objects, the cost matrix C is built over all states, and the assignment remains a bijection between notes and states.
- Lets the optimizer freely trade off modifier usage versus key reuse.
- Scales naturally: adding a new modifier (e.g., Alt) merely enlarges the state list without changing the core algorithm.
- Aligns with the later biomechanical model, where the cost of activating a modifier (e.g., holding Shift) can be expressed as part of the state transition cost.

**Consequences**
- **Positive**: The optimizer now directly optimizes for ergonomic playability, taking modifier cost into account via the C matrix. The same code path supports any number of modifiers by adjusting the keyboard‑state generation step.
- **Negative**: The state space grows linearly with the number of modifier combinations. For the MVP we use two states per base key (plain and shifted), yielding a 2× increase in problem size, which is still trivial for the simulated‑annealing solver.
- **Implementation Impact**:
    - Renamed/clarified `KeyboardState` (was previously a generic term) to hold `modifier: bool` (or enum) alongside the physical key’s geometry and ergonomic metadata.
    - Updated `src/core/kb_builder.load_keyboard_json` to accept a `modifiers` list and generate the Cartesian product of base keys × modifiers.
    - Modified `src/core/objective.Objective` to optionally store `note_labels` and `key_labels` and provide `assignment_dict()` for clean JSON export.
    - Adjusted the CLI (`src/cli/main.py`) to build the state list with modifiers `[False, True]` and to output the assignment via `objective.assignment_dict()`.
    - Updated documentation and ADRs to reflect the change in terminology.
    - Added a new test file (`tests/test_model`) exercising the new API.
