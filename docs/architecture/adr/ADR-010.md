# ADR-010: Calibration procedure for the biomechanical cost matrix C

**Status**: Accepted

**Context**
The matrix C encodes the predicted movement time (or effort) to go from one keyboard state to another.  It depends on several parameters:
- Throughput (TP) – scales the Fitts term.
- Anisotropic distance weights w_x, w_y.
- Finger‑change penalty λ_f, hand‑change penalty λ_h, modifier‑change penalty λ_m.
- Width‑adjustment coefficients κ_w, κ_wf, κ_wh, κ_wm.
- Postural‑effort coefficients α_w (wrist), α_f (finger joints), α_h (humerus), α_t (torso).
- Base actuation cost c_0.
While literature provides reasonable starting guesses, individual anatomies (finger length, joint flexibility, typing skill) can shift these values.  To obtain a user‑specific C that accurately predicts that person’s actual movement times, we devised a short calibration protocol.

**Decision**
Perform a **guided Fitts‑type calibration** consisting of the following blocks (each block contains multiple randomised trials; the participant performs a rapid, accurate tap from a source key to a target key as instructed by on‑screen cues).  After each block we fit the relevant parameters using ordinary least squares (OLS) on the logged movement times (from finger lift‑off to key‑down).  All blocks together take ≈ 12–15 minutes.

| Block | Purpose | Varied Factors | Parameters Estimated |
|-------|---------|----------------|----------------------|
| 1. Baseline Fitts (fixed finger, hand) | Isolate pure distance effect | Horizontal vs vertical distances (Δx, Δy) with same finger/hand, no modifier | TP, w_x, w_y (via slope of MT vs. ID) |
| 2. Finger‑change penalty | Isolate cost of changing finger | Same distance, same hand, differ finger (e.g., index→middle) | λ_f (additive offset in distance-equivalent mm) |
| 3. Hand‑change penalty | Isolate cost of switching hands | Same distance, same finger, differ hand (e.g., left‑index → right‑index) | λ_h |
| 4. Modifier‑change penalty | Isolate cost of engaging/releasing a modifier (e.g., Shift) | Same distance, same finger, same hand, toggle modifier | λ_m |
| 5. Width/angle adjustments | Capture influence of approach angle and effector changes on effective target width | Vary angle (0°, 45°, 90°, 135°) and effector changes (finger‑change, hand‑change, modifier‑change) while keeping distance constant | κ_w, κ_wf, κ_wh, κ_wm (via changes in slope of MT vs. ID) |
| 6. Postural regressions | Estimate joint‑effort coefficients | Random source‑target pairs covering a range of reaches; record wrist deviation, finger‑joint angles (MCP, PIP, DIP), humerus abduction, torso rotation via motion capture or a simple webcam‑based pose estimator. | α_w, α_f, α_h, α_t (via multiple linear regression of MT – (Fitts term + c_0) on joint‑angle magnitudes) |
| 7. Base actuation cost | Measure time to depress and release a key without lateral movement | Finger rests on key, press and release as quickly as possible | c_0 (mean of MT for zero‑distance trials) |

**Procedure Details**
- Equipment: a standard laptop keyboard (the target device), the open‑source software `psychopy` or `PsyToolkit` to present cues and record timestamps via the keyboard’s scan‑code interrupts (millisecond resolution).  Optionally, a webcam or inexpensive motion‑capture system (e.g., OpenPose) for segment 6.
- Each trial: highlight a source key (green) and a target key (red).  Participant keeps a finger on the source, waits for a go‑signal, then moves to tap the target as quickly and accurately as possible.  A timeout of 2 s prevents excessively slow trials.
- After each block, discard trials with movement errors (wrong key or hesitations detected via velocity profile) – typically < 5 % of trials.
- Fit parameters using `scipy.optimize.lsq_linear` (non‑negative least squares where appropriate) or simple linear regression for offset terms.

**Output**: A JSON file `config/calibration.json` containing the estimated values for all parameters listed above.  If the file is absent, the builders fall back to the literature‑based default values documented in `research/06-keyboard-biomechanics.md`.

**Alternatives Considered**
1. **Use population‑average values only** – simpler but fails to capture individual differences; pilot tests showed systematic over‑ or under‑prediction of movement time for participants with atypical finger length.
2. **Full factorial design varying all factors simultaneously** – would require far more trials to achieve statistically stable estimates; the block‑wise approach isolates each factor, improving efficiency.
3. **Bayesian hierarchical modeling** – would provide posterior distributions but adds complexity unnecessary for a point‑estimate calibration step.
4. **Skip calibration and rely solely on analytical derivation from anatomy** – would require precise in‑vivo measurements of finger segment lengths, joint centre locations, and tendon moment arms for each participant, which is impractical in a lab setting.

**Consequences**
- **Positive**: 
    - Produces a C matrix that reflects the actual motor capabilities of the user, improving the ecological validity of the optimisation outcomes.
    - The procedure is quick, requires only a laptop keyboard and optional webcam, and can be repeated whenever the user wishes to update their model (e.g., after a change in typing device).
    - Clear separation: calibration affects only the C builder; the W matrix remains unchanged (it depends solely on tuning and auditory physiology, which are assumed population invariant).
- **Negative**: 
    - Adds a small preprocessing step for users who desire personalization; however, the step is optional—_default parameters still yield a functional system.
    - Relies on the participant’s ability to perform consistent, maximal‑effort taps; fatigue or motivation fluctuations could introduce noise. Mitigated by randomising trial order and discarding outliers.
- **Implementation Impact**: 
    - Added directory `src/calibration/` containing `run_calibration.py` and helper functions for stimulus presentation, timing collection, and OLS fitting.
    - Modified `src/core/kb_builder.py` to first look for `config/calibration.json`; if present, override the default parameters before building C.
    - Updated the top‑level README and the documentation to explain how to run calibration.
